1 Application: A web-based directory of users

The application is a simple directory of users and some related information (name and email). The system has users. There is a special role for some users called Role_Admin.

  1. Viewing the details of a user,
  2. Creating a user,
  3. Updating the details of that user,
  4. Deleting a user.

The rest of the sections have more detailed requirements.

2 User

User is an entity in the system. There are several user(s) in the system whose type is User. The interaction of a user with the system is defined by the requirements below.

2.1 Composition

The system has users. Realized in data model as user-set

2.2 User Identification

Each user has an email address (string that corresponds to a valid email addr).

Realized in data model as cardinality and referential constraints on Email entity.

2.3 Uniqueness of a User

Email identifies a user. Two users with the same email are the same users.

Realized in data model as cardinality and referential constraints on Email entity.

2.4 Name Attribute

2.5 Role of a User

Each user has one or more role, and by default when a user is created, the role that is associated with the newly created user is user

Realized in data model as cardinality and referential constraints on Role entity.

3 Name

Each user has a Name (string).

Realized in data model as cardinality and referential constraints on Name entity.

4 Email

Each user has a Email (string).

Realized in data model as cardinality and referential constraints on Email entity.

5 Role

Different privileges for accessing the system is maintained by roles. A role whose type is Role defines a set of privileges i.e. what is accessible and what is not accessible in the system.

Realized in data model as Role

5.1 Role Mapping to a User

Each user is mapped to exactly one role in a given session, In any given session, a user is mapped to only one of the roles associated with that user

Realized in data model as cardinalility and referential constraints on Role and User entities.

5.2 Types of Roles

There are 2 kinds of roles: admin and user. Each role determines what data access privileges are available to the user.

5.3 Role Privileges

  1. A user with role admin has the ability to:
    1. Create a new user, Read and Update information of all users, and Delete all other users.
  2. A user with role user has the ability to:
    1. View certain information of other users in the system.
    2. Update his/her information.

5.4 User with admin role

There is atleast one user with role admin in the system.

Realized in data model as an admin user

6 Questions/scenarios

Question
Is it possible for the system to have no users? Answer :: No

6.2 System with no admin?

Question
Is it possible for the system to have no user with role admin?
Answer
No.

6.3 System with at least one admin?

Question

Answer
Yes.
